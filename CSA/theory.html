MODULE 1

Platform Overview

What is ServiceNow?

ServiceNow is a cloud-based platform originally built for IT Service Management (ITSM), now extended into HR, Customer Service, Security Operations, and general business workflow automation. The key idea the exam tests: ServiceNow is one platform with many applications built on top of it, all sharing the same database, UI engine, and security model.

Example: The same "Incident" record structure that IT uses to track a broken laptop can be adapted (via tables that extend the Task table) to track an HR case or a facilities request — because every application sits on the same underlying platform.

SaaS vs PaaS — and where ServiceNow fits

ModelWhat the vendor managesWhat you manageExample
IaaSHardware, networking, virtualizationOS, runtime, apps, dataAWS EC2
PaaSHardware + OS + runtimeApplications & dataServiceNow Platform (build custom apps)
SaaSEverything except your data/configConfiguration & data onlyUsing out-of-box Incident Management
Exam Trick: ServiceNow is frequently described as both SaaS and PaaS depending on context — SaaS when you use out-of-box apps as-is, PaaS when you build custom applications on the platform. If a question asks "what type of cloud service model is ServiceNow," the safest exam answer is usually PaaS (because the platform itself is the product being tested), but read the question's framing carefully.

Cloud Concepts ServiceNow Relies On

  • Multi-instance architecture: Each customer gets their own dedicated application instance and database — not a shared (multi-tenant) database. This is different from many other SaaS products.
  • Multi-tenancy at the data center level: Customers share underlying hardware/data center resources, but never share a database with another customer.
  • Automatic upgrades: ServiceNow pushes platform-wide upgrades (twice yearly, named releases e.g. Washington DC, Xanadu) to all instances.
Example: Company A and Company B are both ServiceNow customers. Even though their instances run on shared data center infrastructure, Company A can never query or see Company B's records — each has a logically and physically separate database.

ServiceNow Architecture (High Level)

   ┌─────────────────────────────────────────┐
   │              Browser / Client             │
   └───────────────────┬─────────────────────┘
                        │ HTTPS
   ┌───────────────────▼─────────────────────┐
   │     Application Server (Tomcat-based)     │
   │   - Glide engine, business rules, UI      │
   └───────────────────┬─────────────────────┘
                        │
   ┌───────────────────▼─────────────────────┐
   │          MySQL Database (per instance)    │
   │     Tables, Records, Dictionary, ACLs      │
   └─────────────────────────────────────────┘

Everything a user sees — forms, lists, reports — is generated dynamically by the Glide engine reading metadata (dictionary, ACLs, UI policies) stored in the same database as the actual data.

Instance Types

InstancePurposeWho uses it
Personal Developer Instance (PDI)Free, individual learning/dev sandboxIndividual developers, students
DevelopmentBuild and unit-test new configuration/customizationDevelopers
Test (QA/UAT)Validate changes before they reach real usersTesters, business users (UAT)

Example: A developer builds a new approval workflow in Dev, it's validated in Test by the business team, and only after sign-off is it moved (via an Update Set) into Production where real incidents are processed.
Common Mistake: Assuming you can build directly in Production. ServiceNow's best-practice instance strategy exists specifically to prevent untested changes from breaking the live system.

3 Worked Examples

Example 1
A startup signs up for ServiceNow and only uses the out-of-box Incident Management app without any customization. Which model are they primarily consuming?
Answer: SaaS — they're consuming a ready-made application without building anything new.
Example 2
A company builds a fully custom "Vehicle Fleet Management" application using ServiceNow Studio, custom tables, and workflows.
Answer: PaaS — they are using the platform to build and run a brand-new application.
Example 3
Two unrelated companies both use ServiceNow. Can Company A's administrator ever browse Company B's incident records?
Answer: No — multi-instance architecture means every customer has a logically separate database; there is no shared multi-tenant data layer between customers.
MODULE 2

User Interface

Banner Frame

The bar at the very top of every ServiceNow screen. It contains the company logo, global search, notification bell, the user's profile menu (where you can switch themes, view your profile, and log out), and — depending on configuration — a help icon.

Application Navigator

The collapsible panel (usually on the left) listing every Application the logged-in user has access to, and the Modules (links) inside each application. Clicking a module opens a list or a specific page.

Application Navigator
├── Incident                  (Application)
│   ├── Create New            (Module)
│   ├── Open                  (Module)
│   └── All                   (Module)
├── Service Catalog
│   ├── Catalog Definitions
│   └── Maintain Items
└── System Definition
    ├── Tables
    └── Dictionary
Example: Clicking Incident → Open in the navigator runs a saved filter showing all incidents whose state is not "Closed" or "Resolved."

Favorites & History

  • Favorites: User-curated bookmarks (modules, records, or reports) for quick access — organized into folders.
  • History: An automatically maintained list of the most recently visited pages/records — the user doesn't add to this manually.
Exam Trick: Remember — History is automatic, Favorites is manual. A question describing "the system automatically shows my last 10 visited records" is testing History, not Favorites.

Lists vs Forms

ListForm

Clicking any row in a list opens that record's Form view.

Related Lists

Lists shown at the bottom of a form displaying records from a different table that reference the current record. For example, an Incident form's "Affected CIs" related list shows Configuration Items linked to that incident, even though those CI records live in a different table.

Filter Navigator

The search box at the top of the Application Navigator. Typing a module or application name filters the navigator in real time, letting admins jump straight to a module without manually expanding menus.

Typical UI Navigation Flow

   List View  ──────────────► Personalize List / Filter / Sort / Group By
      │  (click a record row)

Common CSA Questions

Q
What is the difference between a Module and an Application in the navigator?
A: An Application is the top-level grouping (e.g., "Incident"); Modules are the individual links inside it (e.g., "Create New," "Open," "All").
Q
Where would you go to see the 5 records you most recently opened?
A: History — it's automatically tracked, unlike Favorites.
MODULE 3

Users, Groups and Roles

Core Definitions

  • User: A record in the sys_user table representing a person who can log in.
  • Group: A collection of users (sys_user_group), often used for assignment (e.g., "Network Support") rather than direct permission grants.
  • Role: A named permission grant (sys_user_role) that unlocks access to specific data/UI — e.g., itil, admin, catalog_admin.
  • Role Inheritance: Roles can "contain" other roles. If Role A contains Role B, anyone with Role A automatically also has all permissions of Role B.

User → Group → Role Mapping

   User: Priya
      │  is a member of
      ▼
   Group: "Service Desk"
      │  has roles assigned
      ▼
   Roles: itil, catalog_item_viewer
      │  grant access to
      ▼
   ACLs check role → grant/deny read/write on tables & fields

Critically: a User does not need a role directly — they can inherit roles entirely through group membership. This is the most efficient and common way ServiceNow admins manage access at scale.

Example: Instead of assigning the itil role to 200 individual support agents, an admin assigns the role once to the "Service Desk" group, then adds/removes agents from that group as staffing changes.

Role Inheritance Example

RoleContainsNet effect for a user with this role
admin(implicitly) every roleFull system access
itil—Can work with incidents, problems, changes
catalog_admincatalogAnyone with catalog_admin also gets everything the catalog role gives

Scenario-Based Questions

Scenario
John belongs to the "ITIL Users" group, which has the itil role assigned. John personally has no roles assigned on his user record. What access does John get?
Answer: John gets full itil role access, because roles assigned to a group are inherited by every member of that group — a user's own record doesn't need the role directly.
Scenario
Priya is removed from the "Service Desk" group but keeps the itil role directly on her user record. Does she lose ITIL access?
Answer: No — because the role is also assigned directly to her user record, removing her from the group does not remove a directly-assigned role.
Exam Trick: Watch for questions that test whether you know roles can come from two sources simultaneously: directly on the user, and via group membership. Removing one source doesn't remove the other.
MODULE 4

Tables and Databases

Core Definitions

  • Table: A structured collection of records, similar to a spreadsheet (e.g., the Incident table).
  • Record: A single row in a table (one specific incident).
  • Field: A column in a table (e.g., "Priority," "Short Description").
  • Schema: The overall structure — which tables exist and how they relate.
  • Dictionary (sys_dictionary): The metadata table that defines every field on every table — its type, label, length, mandatory status, and more.

Types of Tables

TypeDefinition
Base TableA standalone table not extended from anything (e.g., sys_user).
Parent TableA table that other tables extend from (e.g., Task is parent to Incident).
Child / Extended TableA table built on top of a parent, inheriting all its fields plus adding its own (e.g., Incident extends Task).
Exam Trick: "Parent" and "Child" are relative terms — Incident is a child of Task, but could itself be a parent if something extended it. Don't memorize a table as permanently "parent" or "child"; it depends on the relationship being described.

The Task Table Hierarchy

                    Task (parent table)
                      │
        ┌─────────────┼──────────────┐
        ▼             ▼              ▼
    Incident       Problem      Change Request
   (extends Task) (extends Task) (extends Task)

Because Incident, Problem, and Change Request all extend Task, they automatically inherit Task's common fields: short_description, state, priority, assigned_to, assignment_group, etc. Each child table then adds its own specific fields (e.g., Incident adds caller_id; Change Request adds risk and type).

Example: A Business Rule written on the Task table (e.g., "auto-set urgency based on priority") runs on Incident, Problem, AND Change Request automatically — because they all inherit from Task. This is the single biggest reason ServiceNow uses table extension.

Common Exam Traps

Trap 1: Thinking each child table has a separate copy of the parent's fields. In reality, child table records are stored partly in the parent table and partly in the extension table (via underlying joins) — conceptually, think of it as "inherits," not "duplicates."
Trap 2: Assuming you must redefine common fields like priority on every child table. You don't — define once on the parent (Task), and every extended table gets it for free.
Trap 3: Confusing "Table" with "List view of a table." A list view is just a way of looking at table data; it isn't the table itself.
MODULE 5

Forms and Lists

Configuring Form Layout

Configure Form Layout lets an admin (or user, if permitted) add/remove fields from a form, reorder them, and organize them into sections — without writing any code. Accessed via the gear icon on a form, or right-click → Configure → Form Layout.

  • Sections: Group related fields together (e.g., "Caller Information," "Notes") and can be made collapsible.
  • Related Lists: Configured separately via "Related List" configuration — choosing which related tables show at the bottom of the form.
Example: On the Incident form, an admin uses Configure Form Layout to move the "Category" field next to "Subcategory" and add a new custom field "Customer Impact" into the main section.

Personalizing Lists

Personalize List (also via the gear icon, but on a list) lets a user choose which columns display and in what order — this is a personal preference, not a system-wide change (unless an admin uses "List Layout" administration instead).

ActionWhat it does
FilterShow only records matching conditions (e.g., Priority = 1 - Critical)
SortOrder rows by a column, ascending or descending
Group ByCluster rows under a shared field value (e.g., group incidents by Assignment Group)
Exam Trick: "Personalize List" changes what you see; it does not change the form/list for other users. Confusing personal list changes with global list layout changes is a classic distractor in CSA questions.

MODULE 6

Dictionary and Fields

What is a Dictionary Entry?

Every field on every table has a corresponding record in sys_dictionary defining its type, max length, label, default value, and whether it's mandatory, read-only, or unique. This is the metadata layer that drives how forms render fields automatically.

String›

Definition: Free-text field, single line, with a max character length.

Example: "Short description" on Incident.

Short description: [Email server is down____________]
CSA Question
What happens if a user enters more characters than the String field's max length allows?
A: The input is truncated to the field's defined maximum length.
Integer›

Definition: Whole numbers only, no decimals.

Example: "Reassignment count" on Incident.

CSA Question
Can an Integer field store 4.5?
A: No — Integer fields only store whole numbers; decimals require a Floating Point/Decimal field type.
Choice›

Definition: A dropdown of predefined values, each with a label and an underlying stored value.

Example: "Priority" (1 - Critical, 2 - High, 3 - Moderate, 4 - Low, 5 - Planning).

Priority: [1 - Critical ▼]
CSA Question
Where are the available options for a Choice field defined?
A: In Choice List records (sys_choice) tied to that field and table.
Reference›

Definition: Points to a record in another table — the most important field type for relationships in ServiceNow.

Example: "Assigned to" on Incident references a record in sys_user.

Assigned to: [🔍 Beth Anglin________]  ──► points to sys_user record
CSA Question
What displays in a Reference field on the form — the sys_id or something else?
A: The display value (e.g., user's name) is shown, even though internally the field stores the referenced record's sys_id.
Boolean›

Definition: True/False checkbox.

Example: "Active" field on the User table.

CSA Question
What happens to a user record when "Active" is unchecked?
A: The user can no longer log in, but the record and its history are preserved (not deleted).
Date / Date Time›

Definition: Date stores only a calendar date; Date/Time stores date plus time, automatically adjusted for the viewing user's time zone.

Example: "Opened" (Date/Time) vs a custom "Target completion date" (Date only).

Exam Trick: Date/Time fields are stored in the database in UTC/GMT but displayed converted to each user's personal time zone setting — two users in different time zones see different displayed times for the same underlying record.
HTML›

Definition: Rich text field supporting formatting (bold, bullet points, images) — rendered as HTML.

Example: Knowledge Article body field.

Journal›

Definition: An append-only field — new entries are added with a timestamp and author, and previous entries are never overwritten.

Example: "Additional comments" / "Work notes" on Incident.

Exam Trick: Unlike a String field, you cannot edit or delete a previous Journal entry — each entry is permanently logged, which is exactly why "Work notes" is used for audit trails.
MODULE 7

Service Catalog

Core Definitions

  • Catalog: The overall container (e.g., "Service Catalog," "HR Service Catalog").
  • Category: A grouping within a catalog (e.g., "Hardware," "Software").
  • Catalog Item: An individual requestable thing (e.g., "New Laptop").
  • Record Producer: A catalog-style form that, on submit, creates a record directly in a target table (e.g., Incident) instead of a Request/RITM — useful for letting end users submit incidents through a friendlier catalog-style form.
  • Order Guide: Bundles multiple catalog items together so a user can order a set in one request (e.g., "New Employee Onboarding" bundling a laptop + software + email account).

Visual Hierarchy

Service Catalog
   └── Category: Hardware
          └── Category: Laptops
                 ├── Item: "New Laptop Request"
                 └── Item: "Laptop Repair"
   └── Category: Access
          └── Item: "Software Access Request"

What Happens When You Order an Item

User submits Catalog Item
        │
        ▼
   Request (REQ) created
        │
        ▼
   Requested Item (RITM) created — one per item ordered
        │
        ▼
   Catalog Task(s) (SCTASK) created — fulfillment steps
Example: Ordering "Laptop Request" creates one REQ, one RITM for the laptop, and SCTASKs like "Procure laptop" and "Image and ship laptop" assigned to fulfillment teams.

Scenario-Based Questions

Scenario
A new employee needs a laptop, specific software, and system access all at once. What catalog feature lets them request all three in a single submission?
A: An Order Guide — it bundles multiple catalog items into one guided request.
Scenario
The help desk wants end users to submit incidents through a simplified, catalog-style form rather than the standard Incident form. What should be used?
A: A Record Producer — it creates a record directly in the Incident table from a catalog-style interface.
MODULE 8

Knowledge Management

Core Concepts

  • Knowledge Base (KB): A container for related articles (e.g., "IT Knowledge Base," "HR Policies").
  • Knowledge Article: A single piece of documented content (a how-to, FAQ, or policy) stored in kb_knowledge.

Article Lifecycle

  Draft  ──►  Review  ──►  Published  ──►  Retired
   (author        (approver        (visible to       (no longer
   writes it)     checks it)        end users)         shown)

An article must go through an approval process before it's published — typically requiring a designated approver (often the KB's manager) to sign off. Once published, it can be retired (archived) when outdated, rather than deleted, to preserve audit history.

Example: An agent drafts an article on "How to reset VPN password." It moves to Review where the IT manager approves it. Only after approval does it become visible in the self-service portal as Published.

Common Exam Questions

Q
Can an end user see a Knowledge Article still in "Draft" state?
A: No — only Published articles are visible to general end users; Draft is visible only to the author/contributors.
Q
What's the difference between deleting and retiring an article?
A: Retiring hides the article from active search/browse while preserving its record and history; deleting permanently removes it.
MODULE 9

Flow Designer

Core Concepts

  • Flow Designer: A no-code/low-code visual tool for building process automation — the modern replacement for the legacy Workflow Editor.
  • Trigger: The event that starts a flow — e.g., "Record Created," "Record Updated," "Scheduled," or "Service Catalog."
  • Actions: The individual steps a flow performs — e.g., "Create Record," "Send Notification," "Update Record," "Ask for Approval."
  • Subflow: A reusable mini-flow that can be called from multiple parent flows — useful for logic used in many places (e.g., "Send Standard Notification").

Flow Diagram — When Incident Created

TRIGGER: Record Created (Table: Incident)
        │
        ▼
ACTION: Assign Group  (set assignment_group based on category)
        │
        ▼
ACTION: Send Notification (email assignment group)
        │
        ▼
[Optional] SUBFLOW: "Escalate if Priority 1" (reusable logic)
Example: A flow triggers whenever a new Incident is created with Category = "Network." It automatically assigns the "Network Support" group and emails them — no manual triage needed.

Exam-Oriented Scenarios

Scenario
Three different flows all need to send the same "Approval Granted" email with identical logic. What Flow Designer feature avoids duplicating that logic three times?
A: A Subflow — build the logic once, call it from all three flows.
Scenario
What determines when a flow begins running?
A: Its Trigger — e.g., a record being created, updated, or a schedule being reached.

Reporting

Reports vs Dashboards

  • Report: A single visualization of data from one (or a related) table — a chart, list, or pivot table.
  • Dashboard: A collection of multiple reports/widgets arranged together for an at-a-glance view.

Chart Types

TypeBest forExample
PieProportions of a wholeIncidents by Priority
BarComparing discrete categoriesIncidents per Assignment Group

MODULE 11

Importing Data

Core Concepts

  • Data Source: Defines where the import data comes from (a file upload, an FTP location, a JDBC connection, etc.).
  • Import Set: A staging table where raw imported data first lands, exactly as it appeared in the source — unmodified.
  • Transform Map: The rules that map fields from the Import Set staging table to fields on the real target table, with optional transformation logic.

Import Flow

   Excel / CSV File
        │  (uploaded via Data Source)
        ▼
   Import Set Table   (raw staging data — unchanged)
        │  (Transform Map applies field mapping + logic)
        ▼
   Target Table   (e.g., sys_user, cmdb_ci, custom table)
Exam Trick: The Import Set table is never the final destination — it's always a temporary staging area. If a question asks "where does data live after a successful import," the answer is the Target Table, not the Import Set.

Exam Tips

  • An import can be run manually or scheduled to run automatically on a recurring basis.
  • Transform Maps can include field mapping scripts to perform logic during the transform (e.g., looking up a reference field by name instead of sys_id).
  • You can re-run a Transform Map against the same Import Set data without re-importing the source file.
MODULE 12

Update Sets

What is an Update Set?

An Update Set is a container that captures configuration changes (new fields, business rules, UI policies, ACLs, etc.) made in one instance so they can be moved to another instance — typically Dev → Test → Production. Update Sets move configuration, not transactional data (you wouldn't move actual Incident records this way).

Update Set Lifecycle

DEVELOPMENT INSTANCE
   1. Set Current Update Set as the active "bucket"
   2. Make configuration changes → automatically captured
   3. Mark Update Set "Complete"
        │
        ▼  (Export XML, or use a remote update set connection)
TARGET INSTANCE (e.g., Test/Production)
   4. Retrieve / Import the Update Set
   5. Preview  → checks for collisions with existing customizations
   6. Commit   → applies the changes for real
        │
        ▼ (if something breaks)
   7. Back Out → reverses the committed changes

Key Terms

TermMeaning
CaptureAutomatic — any config change made while an Update Set is "current" is recorded into it
RetrievePulling the Update Set from its source instance into the target instance
PreviewA dry-run check that surfaces conflicts before committing — does NOT apply changes yet
CommitActually applies the captured changes to the target instance
Back OutReverses a previously committed Update Set
Common Exam Trap: Confusing Preview with Commit. Preview only reports potential conflicts — nothing is changed until you Commit. Many CSA questions test whether you know Preview is non-destructive.
Common Exam Trap: Assuming Update Sets capture data records. They do not — Update Sets are for configuration/customization (metadata), never for moving regular business data like actual incidents or users.

Exam-Style Questions

Q
A developer commits an Update Set to Production, but it breaks an existing business rule. What's the fastest way to undo this?
A: Back Out the Update Set — it reverses the committed changes.
Q
Before committing an Update Set, what should always be done first?
A: Preview it, to check for collisions with existing customizations in the target instance.
MODULE 13

Security (ACL)

What is an ACL?

An Access Control List (ACL) is a rule that governs whether a user can perform a specific operation on a table, field, or other resource. Each ACL specifies: the resource it protects, the operation (CRUD), and the condition(s)/role(s) required to pass.

CRUD Operations

LetterOperationExample ACL
CCreateCan this user create a new Incident record?
RReadCan this user view the "Salary" field on HR Case?
UUpdate (Write)Can this user edit the "Priority" field?
DDeleteCan this user delete a Problem record?

How ACLs Evaluate

  User attempts an operation (e.g., Read field "salary")
        │
        ▼
  ServiceNow checks ALL matching ACLs for that table/field/operation
        │
        ▼
  Every matching ACL must pass (role match AND condition script true)
        │
        ▼
  If ANY required ACL fails → operation is DENIED
Exam Trick: ACLs are evaluated with an AND logic across all matching rules — you must pass every applicable ACL, not just one. A common distractor implies "any matching ACL grants access," which is incorrect.

Read/Write Scenarios

Scenario
An ACL on the Incident table's "Read" operation requires the itil role. A user without any roles tries to open an Incident list. What happens?
A: They cannot read/see Incident records — the list will appear empty or access will be denied, since the Read ACL's role requirement isn't met.
Scenario
A field-level ACL restricts Write access on "Approval" to the approver_user role, while the table-level ACL only requires itil. A user has itil but not approver_user. Can they edit the Approval field?
A: No — they pass the table-level ACL but fail the field-level ACL, and since all matching ACLs must pass, the field stays read-only for this user.
MODULE 14

Notifications

Core Concepts

  • Event: A signal fired by the system (often via a Business Rule) indicating something happened — e.g., incident.commented.
  • Notification: A configured record that says "when this event fires (or this record changes in this way), send this message to these recipients."
  • Recipients: Can be specific users, groups, or dynamically determined (e.g., "the record's Assigned to").

Notification Flow

   Record Updated (e.g., Incident state changes to "Resolved")
        │
        ▼
   Event fired (e.g., incident.resolved)
        │
        ▼
   Notification record matches the event/condition
        │
        ▼
   Email sent to configured Recipients (e.g., the Caller)
Example: When an Incident's state changes to "Resolved," a notification automatically emails the Caller letting them know their issue has been addressed.
MODULE 15

Performance Analytics Basics

At the CSA level, you only need foundational awareness of Performance Analytics (PA) — a separate, more advanced trend-analysis application from standard Reporting.

  • Indicator: A specific metric tracked over time (e.g., "% of Incidents resolved within SLA").
  • Widget: A visual display of one or more indicators (a chart, scorecard, etc.).
  • Dashboard: A collection of widgets, same concept as in standard Reporting, but built for trend/historical analysis.
Exam Trick: Know the difference at a high level: standard Reports show a snapshot of current data; Performance Analytics tracks indicators over time using recorded historical snapshots.
